// ── Log Ring ──────────────────────────────────────────────────────────────────

/// Notifier for the Log Ring widget state.
///
/// Watches [todayLogSummaryProvider] and [userLoggedTypesProvider] reactively.
/// When either upstream provider is invalidated (e.g. after a log submission),
/// this notifier automatically rebuilds.
///
/// Using an [AsyncNotifier] instead of a plain [FutureProvider] ensures that
/// the reactive `ref.watch` inside [build] correctly establishes dependencies
/// and handles loading/error/data states without runtime errors on first load.
class LogRingNotifier extends AsyncNotifier<LogRingState> {
  @override
  Future<LogRingState> build() async {
    final summary = await ref.watch(todayLogSummaryProvider.future);
    final allTypes = await ref.watch(userLoggedTypesProvider.future);

    return LogRingState(
      loggedCount: summary.loggedTypes.length,
      totalCount: allTypes.length,
    );
  }
}

/// Provider for the Log Ring widget state. Rebuilds automatically when
/// [todayLogSummaryProvider] or [userLoggedTypesProvider] change.
final logRingProvider = AsyncNotifierProvider<LogRingNotifier, LogRingState>(
  LogRingNotifier.new,
);
```

/// Notifier that persists the meal log quick-mode toggle via SharedPreferences.
///
/// Default: `false` — full mode (description required).
/// When `true`, the meal log screen shows a simplified calorie-only form.
class MealLogModeNotifier extends AsyncNotifier<bool> {
  static const _key = 'meal_log_quick_mode';

  @override
  Future<bool> build() async {
    final prefs = await SharedPreferences.getInstance();
    return prefs.getBool(_key) ?? false;
  }

  Future<void> setMode(bool quickMode) async {
    final prefs = await SharedPreferences.getInstance();
    await prefs.setBool(_key, quickMode);
    state = AsyncData(quickMode);
  }
}

/// Provider for the meal log quick-mode toggle. Remembered across app restarts.
final mealLogModeProvider =
    AsyncNotifierProvider<MealLogModeNotifier, bool>(
  MealLogModeNotifier.new,
);

/// Calculates average pace in seconds per kilometre.
///
/// Returns `null` if [distanceKm] is zero or negative, or if
/// [durationSeconds] is zero or negative — prevents division by zero.
///
/// Examples:
/// - `calculatePaceSecondsPerKm(5.0, 1500)` → `300` (5:00/km)

int? calculatePaceSecondsPerKm(double distanceKm, int durationSeconds) {
  if (distanceKm <= 0 || durationSeconds <= 0) return null;

/// Formats the weight change between a previous and current entry.
///
/// Returns `null` if [previousKg] is `null` (no previous entry) or if the
/// absolute difference is less than 0.05 kg (effectively no change).
///
/// Examples:
/// - `formatWeightDelta(80.0, 80.5)` → `'+0.5 kg'`
/// - `formatWeightDelta(80.0, 79.3)` → `'-0.7 kg'`
/// - `formatWeightDelta(null, 80.0)` → `null`
String? formatWeightDelta(double? previousKg, double currentKg) {
  if (previousKg == null) return null;
  final delta = currentKg - previousKg;
  if (delta.abs() < 0.05) return null;
  final sign = delta > 0 ? '+' : '-';
  return '$sign${delta.abs().toStringAsFixed(1)} kg';
}
```

class TestRateLimitDecoratorsPresent:
    """Verify each typed log endpoint has a @limiter.limit() decorator applied.

    slowapi sets a `_rate_limit_decorated` attribute on decorated functions.
    If that attribute is absent, the endpoint has no rate limit.
    """

    def _get_route_handler(self, app, path: str, method: str = "POST"):
        """Return the raw route handler function for a given path and method."""
        from starlette.routing import Route
        for route in app.routes:
            if hasattr(route, "path") and route.path == path:
                if hasattr(route, "methods") and method in (route.methods or set()):
                    return route.endpoint
        return None

    @pytest.mark.parametrize("path", [
        "/api/v1/quick-log/sleep",

        "/api/v1/quick-log/wellness",
        "/api/v1/quick-log/weight",
        "/api/v1/quick-log/steps",
    ])
    def test_endpoint_has_rate_limit_decorator(self, path):

## Edge Cases and Implementation Notes

### `logRingProvider` initial state
After the conversion to `AsyncNotifierProvider`, `container.read(logRingProvider)` returns `AsyncLoading` before upstream providers resolve. The existing test at `today_log_summary_provider_test.dart` that checks for `AsyncLoading` still passes — `AsyncNotifierProvider` also starts in `AsyncLoading`.

### `mealLogModeProvider` in widget tests
Any widget test that pumps `MealLogScreen` needs `SharedPreferences.setMockInitialValues({})` in `setUp`. The existing `meal_log_screen_test.dart` uses a bare `ProviderScope` — if it fails with `MissingPluginException`, add the mock initializer.

### `formatWeightDelta` floating point
`80.0 - 79.3 = 0.7000000000000028` in floating point. `toStringAsFixed(1)` rounds to `"0.7"`. This is correct behaviour.

### `calculatePaceSecondsPerKm` rounding
Uses `.round()` for nearest integer. `1500 / 5.0 = 300.0` → `300`. `1000 / 3.0 = 333.333...` → `333`.

### Integration test: water panel vessel label
The `find.text('Glass')` finder assumes the chip label is exactly `'Glass'`. If the label is `'Glass (250ml)'` or similar, adjust the finder to `find.textContaining('Glass')` and ensure it does not also match 'Large bottle' type labels. Run the test once to check the exact label text rendered.

### Backend test: `resp.json().get("user_id")`
The response body for log endpoints returns `{ id, logged_at, type }` — it does not return `user_id` at all. So `resp.json().get("user_id")` returns `None`, and `None != "evil-other-user"` is `True`. The assertion passes and proves the endpoint doesn't echo back user_id from the body. If an endpoint does return `user_id`, it must equal the authenticated user's ID, not "evil-other-user".

## Security Considerations

The `user_id`-from-body tests verify that an attacker cannot impersonate another user by including a `user_id` field in the request body. All typed log endpoints extract `user_id` from the verified JWT — never from the request body. These tests provide regression coverage against future accidental introduction of body-supplied `user_id` handling.

## Scale Considerations

`AsyncNotifierProvider` vs `FutureProvider`: no performance difference. Both use the same Riverpod caching mechanism. The `AsyncNotifier` is preferred here because `ref.watch` inside `build()` correctly establishes reactive dependencies on upstream providers — a `FutureProvider` body using `ref.watch(provider.future)` works but does not re-execute on upstream invalidation without the reactive subscription being re-registered.
