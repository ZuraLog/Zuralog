# Architecture Design

## Overview

Zuralog uses a hybrid architecture with two main components that work together:

1. **Cloud Brain** — the backend server that handles data storage, AI orchestration, integration syncing, and background processing
2. **Edge Agent** — the mobile app that handles the user interface, native health platform access (Apple Health and Google Health Connect), and real-time user interaction

The two communicate over a REST API with JWT-based authentication. The Cloud Brain is the intelligence center. The Edge Agent is the sensor and display layer.

## The Hybrid Hub Model

The architecture is designed around one central idea: **all health data from all sources flows through one universal pipeline into one database, and every part of the app reads from the same place.**

This means the Cloud Brain is the hub. Data arrives from many spokes — Strava, Fitbit, Oura, Apple Health, Google Health Connect, manual logs, and any future integration — and all of it passes through the same processing pipeline before being stored. No data is siloed in integration-specific tables. No part of the app has a private data store that other parts cannot see.

The pipeline works in three steps:

1. **Ingest** — raw data arrives from an integration adapter or from the user directly
2. **Store** — every measurement is saved as an event in a single raw event table
3. **Aggregate** — events are deduplicated and crunched into daily summaries that all screens read from

## The Edge Agent (Mobile App)

The Flutter app runs on iOS and Android. It is responsible for:

- **User interface** — all five tabs (Today, Data, Coach, Progress, Trends) and the Settings area
- **Native health access** — reading from Apple HealthKit (iOS) and Google Health Connect (Android) via native platform bridges
- **State management** — local state is managed with Riverpod, navigation with GoRouter
- **Network communication** — all API calls go through a single HTTP client with automatic token injection and silent token refresh

The Edge Agent does not do heavy computation. It does not run AI models, compute correlations, or aggregate data. It sends data to the Cloud Brain and displays what the Cloud Brain sends back.

## The Cloud Brain (Backend Server)

The FastAPI server is the brain of the system. It is responsible for:

- **API layer** — serves all REST endpoints that the mobile app and marketing website consume
- **Integration syncing** — receives webhooks from external services (Strava, Fitbit, etc.) and fetches data from their APIs through integration adapters
- **Data pipeline** — the universal ingest → store → aggregate pipeline that processes all health data
- **AI orchestration** — manages conversations with the AI coach, generates daily insight cards, and runs trend analysis
- **Background processing** — Celery workers handle everything that does not need to happen in real time: syncing, insight generation, notification delivery, health score computation

## Integration Adapter Pattern

Each external data source has an adapter — a small piece of code that translates that source's specific data format into the universal event format. Adding a new integration means writing one adapter. Nothing else in the system needs to change.

This is how the architecture scales to many integrations. The pipeline, the database schema, the aggregation logic, and every screen in the app are completely unaware of how many integrations exist or what their APIs look like.

## Data Flow Summary

```
User action or external webhook
    → Integration Adapter (translates to universal format)
    → Health Events table (raw storage, source of truth)
    → Aggregation Service (dedup + daily totals)
    → Daily Summaries table (read cache)
    → Any screen in the app
```

Every screen reads from the same daily summaries. The Today tab, the Data tab, the Coach, the Progress goals, and the Trends correlations all see the same numbers. There is one version of the truth.

## Authentication

Users authenticate with Supabase Auth (email/password, Google Sign-In, or Apple Sign-In). The mobile app gets a JWT token from Supabase and includes it in every request to the Cloud Brain. The Cloud Brain verifies the token and extracts the user ID. Row-level security at the database level ensures users can only access their own data.

## AI Architecture

The AI layer is not a single model — it is an orchestration system:

- **The Coach** uses a language model (currently via OpenRouter) with the user's health data injected as context
- **Daily Insights** are generated by a pipeline that detects statistical patterns in the data and then uses a language model to write human-readable cards
- **Trend Discovery** is pure statistics (Pearson correlation) with no language model involved — the math finds the patterns, and template text describes them

The AI never invents data. Every observation is traceable to specific numbers in the database.
